## 18. Vault Bootstrap

If you've been pointed to this protocol, your project-specific context should already be loaded — either via your system prompt (Claude Project Instructions) or via the project's guidelines file.

**If you have project context** — you're ready to work. Follow the orientation steps in Section 3.

**If you don't know which project you're working on** — ask the user for the project's home folder path. Store it and never ask again. Then find `0.0 Project Guidelines.md` in that folder.

**If you have NO project context** (e.g., this is a fresh DataWizard deployment with no project set up yet) — the vault needs initial setup.

### First-time vault setup

If the user has not yet pasted the universal project instructions into their Claude Project settings, **instruct them to do so before proceeding.** The instructions block is in the Seed at `_DataWizard/Seed/Project Instructions - Copy-Paste into Claude.md`. Tell the user:

> "Before we get started, you'll need to paste the project instructions into your Claude Project settings. Open `_DataWizard/Seed/Project Instructions - Copy-Paste into Claude.md` in Obsidian and copy the code block into your Claude Project's custom instructions. Once that's done, start a new conversation and I'll be able to orient myself automatically."

For detailed setup guidance, see `_DataWizard/Seed/Guides/Setting Up a Claude Project.md`.

This only needs to happen once per Claude Project. After the instructions are pasted, the LLM will read the protocol and find the project context automatically in every future thread.

### Set up vault backup

Before doing any substantive work in the vault, confirm a backup system is in place. Ask the user:

> "Do you have git backup configured for this vault?"

If yes — confirm it's working and continue.

If no — walk them through setup using the guide at `_DataWizard/Seed/Guides for Humans/Git Backup Setup — Claude-Guided Walkthrough.md` before proceeding. This creates a clean snapshot to roll back to if anything goes wrong during setup or pipeline runs.

### Bootstrapping a new project

Once instructions are pasted, interview the vault owner to create the project infrastructure. A fully bootstrapped project has:

1. **`0.0 Project Guidelines.md`** — the project brief and operational reference. Must include `datawizard_protocol_version` in frontmatter.
2. **`0.1 MOC.md`** — map of contents
3. **`0.2 Session Log.md`** — maintenance diary. Created as a shell file from day one, with a `~Sections/Session Log - Project/` subfolder for individual session entries (see Section 6).
4. **`0.3 Decision Log.md`** — architectural memory. Created at bootstrap — every project makes decisions worth recording.

Additional infrastructure files (`0.4 Harvest Log`, `0.5 Action Items`) are created as the project grows — not all are needed at bootstrap.

### Standard folder structure

A bootstrapped project folder should include:

```
_Project/
+-- 0.0 Project Guidelines - Project.md
+-- 0.1 MOC - Project.md
+-- 0.2 Session Log - Project.md          (shell)
+-- 0.3 Decision Log - Project.md
+-- ! Action Items - Project.md           (when needed)
+-- 1.0 Content Shell.md                  (working docs, as needed)
+-- Resources/                            (supporting materials)
+-- ~Sections/                            (section subfolders)
|   +-- Session Log - Project/
|   +-- Content Shell/
+-- ~Archive - Project/                   (when needed)
```

The `~` prefix sorts utility folders to the bottom of the file explorer. See Section 16 for naming conventions.

### What goes in a 0.0 Project Guidelines file

Interview the vault owner, then draft a guidelines file covering these sections. Adapt depth to project complexity — a lightweight project might only need sections 1, 2, 6, 7, and 8.

1. **What this project is** — one paragraph: name, purpose, scope. If the project is both a shareable system and a live deployment, distinguish the general system from this specific instance.
2. **Current state** — a pointer to the session log and action items. Don't duplicate status here; just direct the reader: "For what's active and what's next, read the last 2-3 session log entries and the action items file."
3. **Core concepts** — the 3-6 ideas, patterns, or design principles that shape how this project works. An incoming instance needs these to make good decisions.
4. **Key architecture decisions** — one-liners pointing to the decision log. Only the decisions that affect how an instance should behave.
5. **Stack / instance-specific config** — hardware, models, tools, key file paths. Label this clearly as instance-specific when the project is a shareable system.
6. **Folder structure** — what's where.
7. **Key pointers** — canonical docs, config files, repos.
8. **Working conventions** — project-specific rules beyond the universal protocol.

The 0.0 should be dense but scannable — aim for 800-1200 tokens so it's cheap to read on every thread orientation. Sections 1 and 3-5 rarely change. Section 2 never goes stale because it points to the session log rather than duplicating status.
